# Security Policy

## Supported Versions

This project is an early-stage public toolkit candidate. Until a stable release process is established, security fixes are expected to apply to the current default branch and the latest tagged release, if one exists.

| Version | Supported |
| --- | --- |
| Latest default branch | Yes |
| Latest tagged release | Yes, if available |
| Older snapshots | No |

## Reporting a Vulnerability

Please do not report suspected vulnerabilities in a public issue. Use GitHub private vulnerability reporting when it is enabled for the repository.

If private vulnerability reporting is not yet enabled, contact the maintainer through the public GitHub profile and ask for a private reporting channel before sharing details.

Include the following information when possible:

- A short description of the issue.
- Steps to reproduce the behavior.
- The affected files, commands, or workflows.
- Any known impact on confidentiality, integrity, or availability.
- Whether the report includes any sensitive data.

## Response Expectations

Maintainers should acknowledge a valid report within five business days, provide an initial assessment, and coordinate a fix before public disclosure when appropriate. If the report is out of scope, maintainers should explain why and suggest a public issue only when it is safe to discuss openly.

## Data Handling

Security reports should avoid including private research content, personal information, credentials, unpublished documents, or confidential institutional material. If sensitive material is required to demonstrate the issue, describe the minimum necessary details and ask maintainers how to share them safely.
